# 10. Composant Header

> **Fichiers concernés**
> - `src/shared/components/Header/Header.tsx`
> - `src/shared/components/Header/Header.module.scss`

---

## Pourquoi ce composant existe

Le Header est le composant transversal le plus important de l'application. Il est présent sur **toutes les pages**, il est le premier élément que l'utilisateur voit, et il conditionne la navigation dans toute l'app.

Il a trois responsabilités distinctes :
1. Afficher le logo (retour à l'accueil)
2. Proposer les **bons liens de navigation** selon qui est connecté
3. Gérer le **responsive** : nav inline sur desktop, hamburger sur mobile

---

## Architecture — le principe de séparation

Le Header est découpé en **deux composants** :

```
Header           ← logique + barre orange
└── MobileMenu   ← drawer mobile (composant séparé)
```

Pourquoi séparer ? Parce qu'un seul fichier qui gère à la fois la barre, le drawer, l'animation, les liens légaux… dépasserait les 150 lignes et mélangerait plusieurs responsabilités. Chaque composant fait une chose.

---

## Le responsive : mobile-first avec CSS

La stratégie est simple : on part du mobile par défaut, et on adapte pour le desktop.

```scss
/* La nav desktop est masquée par défaut (= sur mobile) */
.desktopNav {
    display: none;

    @include desktop {   /* ≥ 1024px */
        display: flex;
    }
}

/* Le bouton hamburger est visible par défaut (= sur mobile) */
.hamburger {
    display: flex;

    @include desktop {
        display: none;
    }
}
```

C'est la mécanique fondamentale du **mobile-first** : on n'écrase pas le mobile depuis le desktop, on construit d'abord le mobile et on enrichit ensuite.

## Le centrage max-width

La barre orange est `width: 100%` (pleine largeur), mais son contenu est limité à `1280px` et centré :

```scss
.header {
    background-color: $color-myOrange;
    width: 100%;             /* fond orange pleine largeur */
    position: sticky;
    top: 0;
}

.inner {
    max-width: $max-width-content;   /* 1280px */
    margin: 0 auto;                  /* centré horizontalement */
    padding: $spacing-xs $spacing-sm;
}
```

**Pourquoi `margin: 0 auto` ?** C'est la technique CSS classique pour centrer un bloc de largeur fixe dans son parent. Le navigateur distribue les marges gauche et droite à égalité. Cela fonctionne parce que `.inner` a un `max-width` défini.

**Résultat sur grand écran (1440px) :** la barre orange s'étend bord à bord, mais le logo et les liens restent dans un couloir de 1280px au centre. Les 80px de chaque côté sont du fond orange vide.

## La navigation dynamique avec `useMemo`

C'est le cœur logique du composant. Les liens de navigation **ne sont pas statiques** — ils dépendent de l'état de connexion et du rôle.

```typescript
const navLinks = useMemo<NavLinkItem[]>(() => {
    if (!isAuthenticated) {
        return [
            { label: 'Accueil', to: '/' },
            { label: 'Se Connecter', to: '/login' },
            { label: "S'inscrire", to: '/inscription' },
        ];
    }

    if (session?.role === 'admin') {
        return [
            { label: 'Accueil', to: '/' },
            { label: 'Administration', to: '/admin' },
            { label: 'Mon profil', to: '/profil' },
        ];
    }

    // client & technicien
    return [
        { label: 'Accueil', to: '/' },
        { label: 'Mes interventions', to: '/interventions' },
        { label: 'Mon profil', to: '/profil' },
    ];
}, [isAuthenticated, session?.role]);
```

**Pourquoi `useMemo` ?** Ce hook mémoïse (met en cache) le résultat du calcul. Sans lui, le tableau `navLinks` serait recréé à chaque re-render du composant, même si `isAuthenticated` et `session?.role` n'ont pas changé. Avec `useMemo`, React ne recalcule le tableau **que si ses dépendances changent**.

**Tableau récapitulatif de navigation :**

| État | Liens affichés |
|---|---|
| Non connecté | Accueil · Se Connecter · S'inscrire |
| Client | Accueil · Mes interventions · Mon profil · Déconnexion |
| Technicien | Accueil · Mes interventions · Mon profil · Déconnexion |
| Admin | Accueil · Administration · Mon profil · Déconnexion |

Ce même tableau `navLinks` est passé au `MobileMenu` — les deux surfaces (desktop nav et mobile drawer) partagent **exactement la même source de vérité**.

## `NavLink` vs `Link` — quelle différence ?

React Router propose deux composants de lien :

- `<Link>` : crée un lien simple. Aucune intelligence sur la route active.
- `<NavLink>` : identique à `<Link>`, mais sait si sa route correspond à l'URL courante.

On utilise `<NavLink>` pour les liens de navigation afin de pouvoir appliquer un style au lien actif :

```typescript
<NavLink
    to={link.to}
    className={({ isActive }) =>
        isActive ? `${styles.navLink} ${styles.active}` : styles.navLink
    }
    end={link.to === '/'}
>
```

**La prop `end`** est importante pour la route `/` : sans elle, tous les liens seraient considérés comme "actifs" sur `/interventions` (car `/` est un préfixe de toutes les URLs). Avec `end`, le match est exact.

## `position: sticky` — le header qui suit le scroll

```scss
.header {
    position: sticky;
    top: 0;
    z-index: 100;
}
```

`sticky` est un hybride entre `relative` et `fixed` :
- L'élément **reste dans le flux** (contrairement à `fixed` qui sort du flux et peut créer des espaces)
- Il "colle" au bord défini (`top: 0`) **dès qu'il atteint ce point au scroll**
- Il est contraint par son parent : il ne dépasse jamais les limites de son conteneur

`z-index: 100` garantit que le header passe par-dessus le contenu de la page au scroll. Le MobileMenu a un `z-index` encore plus élevé (200) pour passer par-dessus le header.

## Accès au contexte d'authentification

```typescript
const { session, isAuthenticated, logout } = useAuth();
```

Le `useAuth()` hook est le point d'accès unique au `AuthContext`. Il expose :
- `session` : `null` si non connecté, sinon `{ userId, role }`
- `isAuthenticated` : booléen raccourci (`session !== null`)
- `logout` : fonction qui vide la session et redirige

La fonction `logout` est passée directement au `<MobileMenu>` via la prop `onLogout`, et au bouton "Déconnexion" de la nav desktop via `onClick`.

---

## Ce que ce composant ne fait PAS

Il est important de noter ce dont le Header **n'est pas responsable** :
- Il ne gère pas la redirection après logout (c'est la responsabilité du `AuthContext`)
- Il ne contient pas les styles du drawer (c'est `MobileMenu`)
- Il ne gère pas l'animation d'ouverture (c'est `MobileMenu`)

Cette séparation des responsabilités rend chaque fichier plus court, plus lisible, et plus facile à tester indépendamment.
