L'authentification est une mécanique qui fait partie de l'application, on va donc intégrer la logique dans `app/providers/authContext`.

# Comment fonctionne l'authentification

## `app/providers/authContext/types/auth.types.ts`

C'est le socle. Avant de coder quoi que ce soit, on définit les types.

Point clé : distinguer `AuthSession` de `User`.

- `AuthSession` = ce que le contexte sait d'une session active : `{ userId, role }`. Pas le token — il est stocké dans un cookie HttpOnly posé par le backend, inaccessible à JavaScript (protection XSS).
- `User` = l'entité profil complète (nom, prénom, téléphone...). Le contexte ne la connaît pas — elle est chargée via API séparément quand un composant en a besoin.

## `app/providers/authContext/auth.context.ts`

`createContext` crée un "canal" React : une boîte vide que le provider remplira avec les vraies valeurs.

Il est séparé du composant provider pour respecter la règle `react-refresh/only-export-components` (nécessaire pour que le Hot Module Replacement fonctionne en dev).

## `app/providers/authContext/AuthContext.tsx`

C'est le composant provider — il remplit la boîte vide créée dans le fichier précédent.

Point technique important :

```ts
const [session, setSession] = useState<AuthSession | null>(readInitialSession);
```

`readInitialSession` est passé comme **initialiseur lazy** de `useState` (sans `()`). React l'appelle une seule fois au montage — c'est plus propre qu'un `useEffect` qui lirait le localStorage après le rendu.

`login()` et `logout()` font deux choses simultanément : mettre à jour l'état React (déclenchant un re-render) ET synchroniser le localStorage (persistance entre rechargements de page). Seuls `{ userId, role }` sont stockés dans localStorage — pas le token.

## `app/providers/authContext/useAuth.ts`

```ts
if (context === undefined) {
    throw new Error('useAuth doit être utilisé dans un AuthProvider');
}
```

Ce guard est essentiel. Sans lui, si un composant appelle `useAuth()` sans être enveloppé par `AuthProvider`, on obtient `undefined` et des bugs silencieux difficiles à tracer. Avec le guard, l'erreur est explicite et immédiate.

# Synthèse

Au rendu du composant `AuthProvider`, on va chercher dans le localStorage si des informations de session existent (`{ userId, role }`). On hydrate l'attribut `value` (`session`, `isAuthenticated`, `isLoading`, `login`, `logout`) de `AuthContext.Provider`, qui communique ces informations à tous ses composants enfants.

Deux événements provoquent le re-render des enfants de `AuthProvider` (soit toute l'application) : un changement de valeur sur les states `session` ou `isLoading`. Les fonctions `login` et `logout` déclenchent ce re-render et synchronisent en même temps le localStorage.

Afin de simplifier la récupération du contexte et de protéger d'une utilisation hors périmètre, on crée un hook `useAuth()` qui récupère le contexte et vérifie que le cadre d'utilisation est correct.
