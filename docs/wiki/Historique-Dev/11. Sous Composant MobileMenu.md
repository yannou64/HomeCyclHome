# 11. Sous-composant MobileMenu

> **Fichiers concernés**

---

## Pourquoi un composant séparé ?

Le `MobileMenu` aurait pu vivre directement dans `Header.tsx`. On l'a isolé pour une raison simple : **il est trop complexe pour être un détail d'implémentation**.

Il gère :
- Une animation CSS de slide
- Un overlay cliquable
- Sa propre structure HTML (logo, liens, séparateur, liens légaux)
- Sa propre logique d'accessibilité

Garder tout ça dans `Header.tsx` aurait produit un fichier de 200+ lignes mélangeant deux problèmes distincts. La règle à retenir : **quand un morceau d'UI a sa propre vie, son propre état et ses propres styles, c'est un composant.**

---

## Le pattern "composant contrôlé"

`MobileMenu` est un composant dit **contrôlé** : il ne gère pas lui-même son état ouvert/fermé. C'est son parent (`Header`) qui possède cet état et le lui communique via les props.

```typescript
type MobileMenuProps = {
    isOpen: boolean;       // Header décide si le menu est ouvert
    onClose: () => void;   // Header fournit la fonction pour fermer
    navLinks: NavLinkItem[];
    onLogout?: () => void;
    isAuthenticated: boolean;
};
```

**Pourquoi ce choix ?** Parce que `Header` est le seul à savoir quand ouvrir le menu (clic hamburger). Si `MobileMenu` gérait son propre état, il faudrait une communication complexe entre les deux. En inversant le contrôle — l'état vit dans le parent, le composant reçoit et obéit — le flux de données reste **unidirectionnel** et prévisible. C'est un principe fondamental de React.

```
Header (possède isMenuOpen)
  │
  ├─ transmet isOpen={isMenuOpen} à MobileMenu
  └─ transmet onClose={() => setIsMenuOpen(false)} à MobileMenu
        │
        MobileMenu n'a aucune idée de comment fonctionne le Header
        Il sait juste : "si on me dit d'être ouvert, je m'ouvre"
```

## L'animation CSS — comment fonctionne le slide

C'est la partie la plus technique. L'animation repose sur **deux propriétés CSS conjuguées** : `transform` et `transition`.

### `transform: translateX()`

Cette propriété **déplace visuellement** un élément sans l'arracher du flux ni déclencher de reflow (recalcul du layout). C'est pour cela qu'elle est préférable à `left: -100%` ou à `margin`.

```scss
.drawer {
    transform: translateX(100%);  /* 100% = largeur du drawer → hors écran à droite */
    transition: transform 0.3s ease;
}

.drawer.open {
    transform: translateX(0);     /* position normale = visible à droite */
}
```

**Pourquoi `100%` et pas une valeur fixe ?** Parce que `100%` est relatif à la largeur de l'élément lui-même (72% du viewport). Si la largeur change (responsive), l'animation reste correcte sans toucher au CSS.

### La classe `.open` comme interrupteur

Le composant est **toujours dans le DOM** (pas de `{isOpen && <nav>}`). La visibilité est contrôlée par la présence ou l'absence de la classe `.open` :

```typescript
<nav className={`${styles.drawer} ${isOpen ? styles.open : ''}`}>
```

**Avantage :** si on utilisait du rendu conditionnel (`{isOpen && ...}`), le composant serait supprimé puis recréé à chaque ouverture — l'animation de fermeture n'aurait pas le temps de se jouer car l'élément disparaîtrait instantanément. Toujours garder l'élément dans le DOM quand on a une animation de sortie.

### L'overlay suit la même logique

```scss
.overlay {
    opacity: 0;
    pointer-events: none;         /* invisible ET non cliquable */
    transition: opacity 0.3s ease;

    &.open {
        opacity: 1;
        pointer-events: auto;     /* visible ET cliquable */
    }
}
```

`pointer-events: none` est crucial : même quand l'overlay est invisible (`opacity: 0`), sans cette propriété il bloquerait tous les clics sous lui.

## Les données statiques vs dynamiques

```typescript
// Données STATIQUES : toujours les mêmes, jamais changées par l'app
const LEGAL_LINKS: NavLinkItem[] = [
    { label: 'Mentions légales', to: '/mentions-legales' },
    { label: 'CGU', to: '/cgu' },
    { label: 'Confidentialité', to: '/confidentialite' },
];

// Données DYNAMIQUES : reçues du parent, changent selon le rôle
export function MobileMenu({ navLinks, ... }: MobileMenuProps) { ... }
```

Les liens légaux sont définis en **constante au niveau du module** (hors du composant). Deux raisons :
1. Ce sont des données qui ne changent jamais → pas besoin de les recalculer à chaque render
2. Les placer dans le composant les ferait recréer à chaque rendu (même résultat, travail inutile)

Les liens de navigation, eux, sont reçus en props car ils **dépendent du rôle** — c'est le `Header` qui les calcule et les partage.

## Accessibilité : les attributs ARIA

```typescript
<nav
    aria-hidden={!isOpen}
    role="dialog"
    aria-label="Menu de navigation"
>
```

- `role="dialog"` : indique aux lecteurs d'écran que cet élément est une fenêtre modale/dialogue
- `aria-hidden={!isOpen}` : quand le menu est fermé, il est masqué aux technologies d'assistance (même s'il est dans le DOM)
- `aria-label` : donne un nom descriptif au dialog (lu par les lecteurs d'écran)

Pour le bouton fermer :
```typescript
<button aria-label="Fermer le menu">
```

Sans `aria-label`, un lecteur d'écran lirait simplement "bouton" (car le contenu du bouton est une image SVG, non lisible).

## Les icônes inline SVG — pourquoi pas un fichier ?

Le hamburger (☰) et le X (×) sont des SVG écrits directement dans le JSX :

```typescript
<svg width="28" height="28" viewBox="0 0 24 24" fill="none"
     stroke="currentColor" strokeWidth="2.5" strokeLinecap="round">
    <line x1="18" y1="6" x2="6" y2="18" />
    <line x1="6" y1="6" x2="18" y2="18" />
</svg>
```

**`stroke="currentColor"` est la clé** : l'icône prend la couleur du texte CSS de son parent (`color: white`). Pas besoin de spécifier la couleur dans le SVG — elle s'adapte automatiquement au contexte.

On aurait pu créer des fichiers `hamburger.svg` et `close.svg` dans les assets. Le choix inline évite deux imports supplémentaires pour des icônes d'une dizaine de lignes. À partir de 3-4 usages du même SVG, un fichier séparé devient préférable.

## Le bouton Déconnexion — rendu conditionnel

```typescript
{isAuthenticated && (
    <li>
        <button onClick={() => { onLogout?.(); onClose(); }}>
            Déconnexion
        </button>
    </li>
)}
```

Deux points à noter :

1. **`isAuthenticated &&`** : le short-circuit evaluation de JavaScript. Si `isAuthenticated` est `false`, React n'évalue même pas ce qui suit et ne rend rien.

2. **`onLogout?.()`** : le `?.` est l'**optional chaining**. `onLogout` est une prop optionnelle (marquée `?` dans le type). Sans `?.`, appeler `onLogout()` quand elle est `undefined` provoquerait une erreur. Avec `?.`, si `onLogout` est `undefined`, l'appel est simplement ignoré.

3. **`onClose()` après `onLogout()`** : on ferme le menu immédiatement après la déconnexion. L'utilisateur ne devrait pas voir le menu rester ouvert pendant la redirection.
