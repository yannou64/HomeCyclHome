# Le hook useLogin

Son rôle : faire le lien entre le service (appel API) et le contexte (mise à jour de la session). Le composant `LoginForm` ne sait pas comment on se connecte — il appelle juste `useLogin()` et reçoit ce dont il a besoin.

```
LoginForm → useLogin → authService → API Backend
                    ↓
               AuthContext (session mise à jour)
```

## Ce que le hook expose au composant

```ts
const { handleLogin, isLoading, error } = useLogin();
```

- `handleLogin(payload)` — déclenche la connexion
- `isLoading` — `true` pendant la requête (pour désactiver le bouton de soumission)
- `error` — message d'erreur si échec (pour l'afficher dans le formulaire)

## Gestion des états

`setError(null)` est appelé au début de chaque tentative pour remettre à zéro l'erreur précédente.

`setIsLoading(false)` est dans le bloc `finally` — il s'exécute toujours, que la requête réussisse ou échoue. Sans `finally`, un échec laisserait le bouton désactivé indéfiniment.

## Renommage de l'import

```ts
import { login as loginService } from '../services/authService';
```

`login` existe déjà dans le hook (récupéré depuis `useAuth`). Le renommage évite le conflit de noms.

## Séparation des responsabilités

Le hook ne connaît pas l'UI — il ne sait pas comment le formulaire est construit.
Le composant ne connaît pas la logique — il ne sait pas comment on appelle l'API.
Chacun a une responsabilité unique.
